Flujo completo — Plataforma Agentes Cleexs (Teo)

Producto: Centro de Gestión de Agentes Cleexs  |  Primer agente: Teo (SEO / AEO / contenido)  |  Primer workspace: Cleexs  |  Versión: 1.0 MVP  |  Fecha: Julio 2026

Principio rector: El agente trabaja de forma autónoma. El cliente configura temas y reglas al inicio; el backoffice sirve principalmente para medir resultados y, cuando corresponde, aprobar piezas puntuales.

1. Visión general

La plataforma Agentes Cleexs es un sistema multiagente separado del producto PRIA (app.cleexs.net). Cleexs es el primer cliente (workspace); Teo es el primer agente.

ProductoRol
Cleexs (PRIA)Mide y diagnostica visibilidad
Plataforma Agentes CleexsProduce, publica y mantiene contenido
cleexs.net (WordPress)Sitio público — blog / artículos
Centro de Gestión (backoffice)El cliente configura poco y mide resultados

No se usa en esta plataforma: n8n, LangChain (fase inicial), vocabulario PRIA (runs, diagnósticos).

5. Flujo automático completo

Fase A — Configuración (poco, al inicio)

  • Temas / keywords prioritarios
  • Tono de voz y entidad (marca, productos)
  • Tipos de pieza permitidos
  • Frecuencia (ej. 2/semana)
  • Integraciones: WordPress, GA4, GSC
  • ¿Requiere aprobación humana? (sí en MVP)

Fase B — Teo autónomo (sin intervención)

  1. Scheduler revisa workspaces con temas configurados
  2. Si no hay misión activa y pasó el intervalo → crea misión
  3. Estratega elige tema + tipo de pieza
  4. Researcher → outline + fuentes
  5. Escritor → borrador
  6. Albañil SEO → schema, meta, slug, links
  7. Se crea ContentPiece en base de datos

Fase C — Aprobación (si autoPublish = false)

  • Pieza aparece en cola del backoffice
  • Cliente revisa / edita opcionalmente
  • Aprueba o rechaza

Fase D — Publicación en WordPress

  • API REST → crea post (draft o publish)
  • Guarda externalId y URL en Publication
  • Registra actividad en el feed

Fase E — Medición y refresco (continuo)

  • Sync GSC/GA4 por URL (cron diario)
  • Dashboard de Resultados
  • Si pieza envejece o cae → Refrescador propone update → nueva misión
FLUJO RESUMIDO: CONFIG (temas, tono, reglas) │ ▼ SCHEDULER (cron interno) │ ▼ MISIÓN → Estratega → Researcher → Escritor → Albañil SEO │ ├── autoPublish=false → Cola de Aprobación → Cliente aprueba │ │ └── autoPublish=true ──────────────────────────┤ ▼ WordPress (draft/publish) │ ▼ Métricas GSC/GA4 │ ▼ Refresco si hace falta

6. Quién decide qué publicar

Hoy (MVP)

  • Rota entre temas configurados
  • Alterna tipos de pieza (FAQ, comparativa, checklist, how-to, pilar)
  • Intervalo mínimo: ~3 días entre misiones por workspace

Fase 2 (con GSC/GA4)

  • Prioriza URLs con caída de impresiones
  • Detecta gaps vs competencia
  • Refresca piezas viejas automáticamente

El cliente no elige cada artículo. Define el territorio (temas); Teo opera dentro de eso.

12. Roadmap

Fase 1 — MVP (actual)

  • Monorepo, API, executor, scheduler, WordPress REST, backoffice
  • Pendiente: DB en Easypanel, flujo completo, primer ecosistema real

Fase 2 — Contenido y métricas

  • Plantillas HTML, permalinks, GSC/GA4, Resultados reales, Refrescador

Fase 3 — Escala

  • Empleados como workspace 2, Periodista, Contactador, LangChain si hace falta, segundo agente
Resumen ejecutivo: El cliente configura temas y reglas una vez. Teo trabaja solo: planifica, investiga, escribe, optimiza SEO y deja la pieza lista. Si hay aprobación humana, el cliente revisa en minutos y al aprobar se publica en WordPress. El backoffice sirve principalmente para medir resultados. Cleexs es el primer caso de éxito; la plataforma está diseñada para múltiples agentes y múltiples clientes.
